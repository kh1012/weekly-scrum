# Weekly Snapshot v1 포맷 가이드

본 문서는 **위클리 스크럼 시스템에서 사용하는 공식 스냅샷 입력 포맷(v1)**을 정의한다.  
팀원은 이 문서에 따라 매주 Snapshot을 작성하며, 이 데이터는 팀/개인의 대시보드, 협업 분석, 리스크 탐지 등에 활용된다.

아래는 실제 작성되는 스냅샷의 예시다.

```
[Frontend / MOTIIV / Spreadsheet / 셀 렌더링 개선]
- Name: 김서연
- Plan: 셀 최적화 100% 완료
- Progress: 셀 렌더링 구조화 및 개선 작업 100% 완료
- Next: 렌더링 최적화 마무리 및 저장 테스트 추가
- Risk: 저장 플로우에서 race condition 재발 가능성 확인
- RiskLevel: 1
- Collaborators: 김정빈(pair), 조해용(waiting-on)
```

---

# 1. 스냅샷 헤더 구조 v1

스냅샷의 첫 줄은 아래와 같은 4단계 구조로 이루어진다.

```
[domain / project / module / topic]
```

아래는 v1 확정 구조다.

---

## 1.1 domain (업무 관점의 최상위 분류)

팀원이 어떤 역할/관점에서 일을 했는지 나타낸다.

- Planning (기획/정보구조/요구정의)
- Design (UI/UX)
- Frontend
- Backend
- Operation (팀운영/온보딩/운영/지원)
- Collaboration (유관부서 협업)
- Content (아티클/시트 제작 등)
- Research (R&D)

_역할이라기보다 “이번 일을 어떤 관점에서 수행했는가”를 표기하는 영역이다._

---

## 1.2 project (업무가 속한 실질적 프로젝트 명)

현재 팀이 운영하는 주요 프로젝트는 다음과 같다.

- MOTIIV
- M-Connector
- M-Desk
- Idea-forge
- …

향후 프로젝트가 추가될 수 있으므로 확장 가능 구조이다.

---

## 1.3 module (프로젝트 내부의 하위 영역)

project 아래에서 더 좁혀진 기능 단위 또는 업무 범위를 나타낸다.

예: MOTIIV 기준

- 워크스페이스
- 팀프로젝트
- 스프레드시트
- 홈
- 프로필
- 뱃지
- 알림
- …

module은 **MOTIIV 한정이 아니며**,  
다른 프로젝트에서도 필요하면 자유롭게 정의할 수 있는 개방형 구조다.

---

## 1.4 topic (구체적인 일의 이름)

- 자유롭게 정의하며 하나의 스냅샷을 대표하는 주제 제목이다.
- 예:
  - 셀 렌더링 개선
  - Import 구조 개선
  - 권한 모델 정비
  - 온보딩 문서 개편

---

# 2. 전체 스냅샷 포맷 (예시 포함)

아래는 v1 기준으로 확정된 스냅샷 텍스트 포맷이다.

```
[Frontend / MOTIIV / Spreadsheet / 셀 렌더링 개선]
- Name: 김서연
- Plan: 셀 최적화 100% 완료
- Progress: 셀 렌더링 구조화 및 개선 작업 100% 완료
- Next: 렌더링 최적화 마무리 및 저장 테스트 추가
- Risk: 저장 플로우에서 race condition 재발 가능성 확인
- RiskLevel: 1
- Collaborators: 김정빈(pair), 조해용(waiting-on)
```

---

# 3. 본문 필드 정의

## 3.1 Name

작업을 수행한 팀원의 이름

---

## 3.2 Plan

이번 주에 처음 설정했던 “목표/계획”
반드시 Percent(%)가 포함되어야 함.

- 무엇을 어느 정도까지 하려 했는가
- 목적 중심으로 간단 명확하게 작성

---

## 3.3 Progress

실제로 수행한 결과
반드시 Percent(%)가 포함되어야 함.

- Plan과 Progress의 차이를 통해 달성률과 병목을 분석 가능

---

## 3.4 Next

다음 주 혹은 다음 단계에서 이어서 수행해야 할 작업

---

## 3.5 Risk

현재 작업 진행에서 발생한 리스크 또는 잠재적 문제 요소

예:

- 저장 플로우에서 race condition 재발 가능성
- 특정 브라우저 레이아웃 깨짐 위험
- API 응답 지연 가능성

없으면 생략 가능

---

## 3.6 RiskLevel

리스크의 강도/중요도를 0–3 사이 값으로 표시한다.

| Level | 의미                            |
| ----- | ------------------------------- |
| 0     | 없음 / 영향 낮음                |
| 1     | 경미, 인지 필요                 |
| 2     | 주의 필요, 후속조치 요구        |
| 3     | 심각, 리소스/우선순위 조정 필요 |

---

# 4. Collaborators 필드

```
- Collaborators: 김정빈(pair), 조해용(waiting-on)
```

## 4.1 기본 형식

```
이름(relation)
```

여러 명은 쉼표로 구분한다.

---

## 4.2 relation 타입 상세 정의

relation은 이번 주 작업 흐름에서 **협업이 어떻게 연결되었는지**,  
그리고 **어디에서 자연스럽지 않은 멈춤이 있었는지**를 기록하기 위한 데이터이다.  

relation은 상태를 평가하거나 책임을 표현하기 위한 항목이 아니라,  
**팀의 실제 협업 흐름을 관찰하여 패턴을 이해하기 위한 로그(Log)**로 사용한다.

relation에는 아래 2가지 타입을 사용한다.

---

## pair

> **두 역할이 동일한 작업 흐름을 실시간 조율하며 함께 진행한 경우 기록한다.**  
> 혼자 수행하기 어려운 작업을 상호 의존적으로 풀어나가는 상황을 의미한다.

### ■ 판단 기준
아래 중 하나라도 해당하면 pair로 기록한다.

1. 작업이 두 역할의 **동시 조율**을 필요로 한다.  
2. 한쪽이 끝내고 넘기는 구조가 아니라, **실시간 상호작용**이 작업의 본질이다.  
3. 협업이 있어야만 작업이 성립하며, 함께 만든 결과물이 명확하다.

### ■ 대표 사례
- FE–BE가 스펙 및 API 인터페이스를 실시간 협의한 경우  
- 기획–디자인이 화면 흐름과 UX를 동시에 조율하며 시안을 구성한 경우  
- 문제 해결 과정 전체가 둘의 공동 판단과 작업으로 구성된 경우  

### ■ 기록 목적
- 실제 협업 강도와 기여 패턴을 드러내기 위함  
- 특정 작업이 어떤 역할 간 협업을 필요로 하는지 구조적으로 이해하기 위함  

---

## waiting-on

> **작업 흐름이 자연스럽게 이어지지 않고,  
> 다음 단계로 연결되기 전 잠시 멈춤이 있었던 구간을 기록한다.**  
> 이 멈춤은 보통 다른 역할의 준비·확정·결정이 완료된 후 다시 흐름이 이어지는 구조에서 발생한다.

### ■ 판단 기준
waiting-on은 모든 연계를 기록하는 항목이 아니다.  
아래 기준에 해당될 때만 기록한다.

1. 다음 단계로 진행하기 위해 필요한 준비·확정·결정이 잠시 머물러 있었던 경우  
2. 이번 주 과정에서 작업 흐름이 **일시적으로 이어지지 않고 멈춘 지점**이 존재했을 경우  
3. 내가 이어서 진행할 수 있는 작업이 모두 소진되어,  
   다음 단계가 준비될 때까지 흐름이 잠시 머물렀던 경우  

### ■ 다음 주로 이어지는 과제의 waiting-on
과제가 다음 주까지 이어지는 경우에도,  
**이번 주 작업 흐름에서 잠시 멈춤이 있었던 시점이 존재했다면 waiting-on으로 반드시 기록한다.**  
waiting-on은 완료 여부와 무관하게 **이번 주 흐름의 상태를 기록하는 협업 로그**이기 때문이다.

### ■ 대표 사례
(부드러운 표현으로 나열)

- 스펙·정책·시안 등 다음 단계로 이어지기 위한 준비가 잠시 보류된 경우  
- 검토·결정·확정 과정이 예상보다 길어져 흐름이 자연스럽게 이어지지 못한 경우  
- 기능 구현 또는 테스트가 다음 단계 준비를 기다리며 잠시 멈춰 있었던 경우  

### ■ 기록 목적
waiting-on은 책임이나 지연을 판단하기 위한 항목이 아니다.  
**작업 흐름이 어떻게 이어졌고, 어디에서 자연스럽게 이어지지 않았는지 기록하여  
협업 패턴과 구조적인 흐름을 파악하기 위함**이다.

---

## 기록 시점 (가장 중요한 규칙)

### ✔ relation은 **이번 주 실제로 발생한 협업 흐름(Realized Collaboration)**만 기록한다.
- pair → 이번 주에 실제로 함께 진행한 협업  
- waiting-on → 이번 주에 실제로 흐름이 잠시 멈춘 구간  

### ✔ 다음 주에 필요할 협업은 relation에 기록하지 않는다.
- 다음 주 계획은 **next 섹션**에서만 텍스트로 기술한다.  

---

## pair vs waiting-on 경계

| relation       | 의미        | 핵심 기준 |
|----------------|-------------|------------|
| **pair**       | 협업 중     | 두 역할이 실시간 상호작용으로 함께 작업이 전진함 |
| **waiting-on** | 연결 대기   | 다음 단계로 이어지기 전에 잠시 멈춤이 있었던 지점 |

---

## 핵심 요약

- **pair** → 함께 풀어간 협업  
- **waiting-on** → 흐름이 자연스럽게 이어지지 않고 잠시 머물렀던 지점  
- 둘 다 책임이 아니라, **흐름을 기록하는 관찰 데이터**  
- 이번 주 실제 흐름만 기록  
- 다음 주 계획은 next에만 작성  
- 과제가 연속되더라도 이번 주의 멈춤은 반드시 기록

# 5. 스냅샷 구조와 대시보드 활용

| 요소          | 활용 예                           |
| ------------- | --------------------------------- |
| domain        | 리소스/역량 분포 분석             |
| project       | 프로젝트별 업무 흐름 파악         |
| module        | 특정 기능 영역의 집중도·병목 파악 |
| topic         | 구체적인 업무 추적                |
| collaborators | 협업 패턴, 블로킹 관계, 병목 파악 |

해당 구조는 이후 AI 요약, 협업 패턴 분석, 임팩트 분석 등의 기반 데이터로 활용된다.

---

# 6. 요약

스냅샷 v1 포맷은 다음 네 가지 흐름을 기반으로 한다:

1. **domain → project → module → topic**  
   — 일의 위치를 계층적으로 구조화

2. **Plan → Progress → Next → Risk → RiskLevel**  
   — 실행 상황을 한눈에 파악하기 위한 핵심 필드

3. **Collaborators(name(relation))**  
   — 협업 및 의존 관계를 최소 입력으로 명확하게 표현

4. **확장성과 일관성을 모두 고려한 구조**  
   — 프로젝트, 모듈, relation 타입이 계속 추가될 수 있는 형태
