# Weekly Snapshot v1 포맷 가이드

본 문서는 위클리 스크럼 시스템에서 사용하는 공식 스냅샷 입력 포맷(v1)을 정의한다.  
팀원은 이 문서에 따라 매주 Snapshot을 작성하며, 이 데이터는 팀/개인의 대시보드, 협업 분석, 리스크 탐지 등에 활용된다.

아래는 실제 작성되는 스냅샷의 예시다.

[Frontend / MOTIIV / Spreadsheet / 셀 렌더링 개선]
- Name: 김서연
- Plan: 셀 최적화 100% 완료
- Progress: 셀 렌더링 구조화 및 개선 작업 100% 완료
- Next: 렌더링 최적화 마무리 및 저장 테스트 추가
- Risk: 저장 플로우에서 race condition 재발 가능성 확인
- RiskLevel: 1
- Collaborators: 김정빈(pair), 조해용(pre)


# 1. 스냅샷 헤더 구조 v1

스냅샷의 첫 줄은 아래와 같은 4단계 구조로 이루어진다.

[domain / project / module / topic]


## 1.1 domain (업무 관점의 최상위 분류)

domain은 "이번 일을 어떤 **관점**에서 수행했는가"를 나타내며,  
직무(job title)가 아니라 **작업의 성격**을 반영한다.  
같은 사람이 주차별로 다양한 domain에서 작업할 수 있다.

아래는 각 domain의 의미와 사용 기준이다.

- **Planning**  
  기획·정보 구조·요구 정의·정책 설계 등  
  “무엇을 만들 것인가”, “어떤 규칙으로 동작해야 하는가”를 정의하는 관점.  
  예: IA 개편, 기능 요구사항 작성, 정책 흐름 정리

- **Design**  
  UX/UI·화면 구조·사용자 경험 설계 등  
  서비스 경험을 시각적·구조적으로 구체화하는 관점.  
  예: UI 시안, UX 플로우 정리, 인터랙션 설계

- **Frontend**  
  클라이언트 로직 구현·UI 개발·상태 관리·최적화 등  
  사용자 인터페이스를 실제 코드로 구현하는 관점.  
  예: 컴포넌트 개발, API 연동, 렌더링 최적화

- **Backend**  
  서버 로직·데이터 모델링·API 개발·인프라 운영 등  
  시스템 내부의 데이터·비즈니스 로직 중심의 관점.  
  예: API 개발, DB 스키마 정의, 트랜잭션 로직 구현

- **Operation**  
  팀 운영·프로세스 정비·온보딩·문서화·운영지원 등  
  프로젝트 수행을 돕는 운영 중심의 작업.  
  예: 온보딩 문서 제작, 운영 정책 정비, 회의 기록/정리

- **Collaboration**  
  유관부서와의 공동 작업, 조율, 협업 중심의 관점.  
  단순 전달이 아니라 협업 그 자체가 핵심일 때 사용.  
  예: 영업/마케팅/CS 협업, 외부 파트너 대응

- **Content**  
  문서·시트·튜토리얼·아티클 등 콘텐츠 제작 중심의 작업.  
  예: 실습 시트 제작, 교육자료 작성, 가이드 문서 제작

- **Research**  
  실험·파일럿 검증·기술 탐색 등  
  결과보다 탐구·검증·실험이 핵심일 때 사용.  
  예: 신규 라이브러리 테스트, UX 실험, 프로토타입 제작


## 1.2 project (업무가 속한 실질적 프로젝트)

- MOTIIV  
- M-Connector  
- M-Desk  
- Idea-forge  
- … (확장 가능)

## 1.3 module (프로젝트 내부 하위 영역)

예: MOTIIV 기준  
- 워크스페이스  
- 팀프로젝트  
- 스프레드시트  
- 홈  
- 프로필  
- 뱃지  
- 알림  

각 프로젝트별로 자유롭게 정의한다.

## 1.4 topic (구체적인 일의 이름)

스냅샷을 대표하는 핵심 작업 주제.

예:  
- 셀 렌더링 개선  
- Import 구조 개선  
- 권한 모델 정비  
- 온보딩 문서 개편

- Plan: 셀 최적화 100% 완료
- Progress: 셀 렌더링 구조화 및 개선 작업 100% 완료
- Next: 렌더링 최적화 마무리 및 저장 테스트 추가
- Risk: 저장 플로우에서 race condition 재발 가능성 확인

# 3. 본문 필드 정의

## 3.1 Name
작업을 수행한 팀원의 이름.

## 3.2 Plan
이번 주 시작 시점에 설정한 목표. 반드시 Percent(%) 포함.  
목적 중심·결과 중심으로 간결하게 작성.

## 3.3 Progress
이번 주 실제 수행 결과. 반드시 Percent(%) 포함.  
Plan과 Progress의 차이를 기반으로 병목, 리스크, 진행도 판단이 가능하다.

## 3.4 Next
다음 주 또는 다음 단계에서 이어서 진행할 작업.

## 3.5 Risk
작업 과정에서 발생한 리스크 또는 잠재적 문제 요소.  
없으면 생략 가능.

예:  
- race condition 가능성  
- 브라우저 레이아웃 깨짐 위험  
- API 응답 지연 우려

## 3.6 RiskLevel

| Level | 의미 |
| ----- | ----- |
| 0 | 없음 / 영향 낮음 |
| 1 | 경미, 인지 필요 |
| 2 | 주의 필요, 후속 조치 요구 |
| 3 | 심각, 리소스·우선순위 조정 필요 |


# 4. Collaborators 필드

Collaborators는 **이번 주 실제 협업 흐름을 기록하는 항목**이다.

형식은 다음과 같다.

이름(relation), 이름(relation)

실제 스냅샷 기록에서는 relation을 다음 3개 값만 사용한다.

pair  
pre  
post  

단, 설명에서는 (pair partner), (pre partner), (post partner) 용어로 보조 설명을 제공한다.


# 4.1 relation 타입 상세 정의


## pair (pair partner)

두 역할이 **동시에 공동 작업을 수행한 관계**.

### 판단 기준
- 실시간 상호작용이 작업의 본질  
- 한쪽이 끝내고 넘기는 구조가 아님  
- 둘의 판단과 작업이 결과물에 동시에 반영됨

### 대표 사례
- FE–BE가 API 스펙을 실시간으로 조율  
- 기획–디자인이 UX 흐름을 함께 구성  
- 퍼포먼스 튜닝을 함께 분석하며 해결

### 기록 목적
- 협업 강도·기여 패턴·역할 조율 패턴을 파악  


## pre (pre partner)

내 작업의 **앞단에서 입력·결정·맥락을 제공한 협업 관계**.

### 판단 기준
- 내 작업이 진행되기 위해 선행 정보/확정/시안/정책이 필요  
- 이번 주 흐름에서 실제로 영향을 주었음  
- 선행 영향이 명확한 경우 기록

### 대표 사례
- 디자인 시안 전달 이후 FE 작업 시작  
- 정책/기획 확정 이후 기능 개발 착수  
- BE 스키마 확정 이후 FE API 연동 진행

### 기록 목적
- 작업이 어디에서 출발하고 어떤 선행 요소가 영향을 주었는지 파악  


## post (post partner)

내 작업 결과가 **다음 단계에서 사용되거나 영향을 주는 관계**.

### 판단 기준
- 내 산출물이 다음 역할의 작업을 구성  
- 이번 주 실제 흐름에서 전달·사용됨  

### 대표 사례
- FE 산출물을 QA가 검증 시작  
- 기획 산출물을 디자인이 이어받아 상세 시안 제작  
- API 개발 결과를 FE가 연동

### 기록 목적
- 작업 결과가 어떤 흐름으로 이어졌는지 파악  
- 다음 단계 병목 탐지  


# 5. relation 기록 규칙

## 이번 주 실제로 발생한 흐름만 기록
- pair → 이번 주 공동 협업  
- pre → 이번 주 내 작업의 앞단 영향  
- post → 이번 주 내 결과가 다음 단계로 이어짐  

## 다음 주 필요한 협업은 기록하지 않음
다음 주 협업 예상은 relation이 아니라 Next에만 작성.

## 과제가 연속되더라도
이번 주 실제 흐름에서 pre 또는 post 관계가 존재했다면 반드시 기록한다.


# 6. relation 요약

| relation | 설명 | 기준 |
|---------|------|-------|
| pair | 실시간 공동 협업 (pair partner) | 둘의 동시 상호작용이 핵심 |
| pre | 앞단 협업자 (pre partner) | 내 작업에 필요한 선행 입력 제공 |
| post | 후단 협업자 (post partner) | 내 결과물을 받아 다음 단계 수행 |


# 7. 스냅샷 데이터 활용

| 요소 | 활용 |
|------|------|
| domain | 역할/역량 분포 분석 |
| project | 프로젝트별 흐름 파악 |
| module | 기능 단위 집중도·병목 분석 |
| topic | 개별 작업 추적 |
| collaborators | 협업 패턴 분석, 블로킹 요소 탐지, 흐름 구조 이해 |


# 8. 요약

스냅샷 v1 포맷은 다음 원칙을 따른다.

1. domain → project → module → topic  
2. Plan → Progress → Next → Risk → RiskLevel  
3. Collaborators: 이름(pair / pre / post)  
4. relation 설명은 문서 내에서만 (pair partner), (pre partner), (post partner)로 표기  
5. 실제 협업 흐름을 기반으로 팀의 구조적 패턴을 이해하는 것이 목적  

본 가이드는 파서, 대시보드, AI 분석 기준에서 동일하게 적용된다.
