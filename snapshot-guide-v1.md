# Weekly Snapshot v1 포맷 가이드

본 문서는 **위클리 스크럼 시스템에서 사용하는 공식 스냅샷 입력 포맷(v1)**을 정의한다.  
팀원은 이 문서에 따라 매주 Snapshot을 작성하며, 이 데이터는 팀/개인의 대시보드, 협업 분석, 리스크 탐지 등에 활용된다.

아래는 실제 작성되는 스냅샷의 예시다.

```
[Frontend / MOTIIV / Spreadsheet / 셀 렌더링 개선]
- Name: 김서연
- Plan: 셀 최적화 100% 완료
- Progress: 셀 렌더링 구조화 및 개선 작업 100% 완료
- Next: 렌더링 최적화 마무리 및 저장 테스트 추가
- Risk: 저장 플로우에서 race condition 재발 가능성 확인
- RiskLevel: 1
- Collaborators: 김정빈(pair), 조해용(waiting-on)
```

---

# 1. 스냅샷 헤더 구조 v1

스냅샷의 첫 줄은 아래와 같은 4단계 구조로 이루어진다.

```
[domain / project / module / topic]
```

아래는 v1 확정 구조다.

---

## 1.1 domain (업무 관점의 최상위 분류)

팀원이 어떤 역할/관점에서 일을 했는지 나타낸다.

- Planning (기획/정보구조/요구정의)
- Design (UI/UX)
- Frontend
- Backend
- Operation (팀운영/온보딩/운영/지원)
- Collaboration (유관부서 협업)
- Content (아티클/시트 제작 등)
- Research (R&D)

_역할이라기보다 “이번 일을 어떤 관점에서 수행했는가”를 표기하는 영역이다._

---

## 1.2 project (업무가 속한 실질적 프로젝트 명)

현재 팀이 운영하는 주요 프로젝트는 다음과 같다.

- MOTIIV
- M-Connector
- M-Desk
- Idea-forge
- …

향후 프로젝트가 추가될 수 있으므로 확장 가능 구조이다.

---

## 1.3 module (프로젝트 내부의 하위 영역)

project 아래에서 더 좁혀진 기능 단위 또는 업무 범위를 나타낸다.

예: MOTIIV 기준

- 워크스페이스
- 팀프로젝트
- 스프레드시트
- 홈
- 프로필
- 뱃지
- 알림
- …

module은 **MOTIIV 한정이 아니며**,  
다른 프로젝트에서도 필요하면 자유롭게 정의할 수 있는 개방형 구조다.

---

## 1.4 topic (구체적인 일의 이름)

- 자유롭게 정의하며 하나의 스냅샷을 대표하는 주제 제목이다.
- 예:
  - 셀 렌더링 개선
  - Import 구조 개선
  - 권한 모델 정비
  - 온보딩 문서 개편

---

# 2. 전체 스냅샷 포맷 (예시 포함)

아래는 v1 기준으로 확정된 스냅샷 텍스트 포맷이다.

```
[Frontend / MOTIIV / Spreadsheet / 셀 렌더링 개선]
- Name: 김서연
- Plan: 셀 최적화 100% 완료
- Progress: 셀 렌더링 구조화 및 개선 작업 100% 완료
- Next: 렌더링 최적화 마무리 및 저장 테스트 추가
- Risk: 저장 플로우에서 race condition 재발 가능성 확인
- RiskLevel: 1
- Collaborators: 김정빈(pair), 조해용(waiting-on)
```

---

# 3. 본문 필드 정의

## 3.1 Name

작업을 수행한 팀원의 이름

---

## 3.2 Plan

이번 주에 처음 설정했던 “목표/계획”
반드시 Percent(%)가 포함되어야 함.

- 무엇을 어느 정도까지 하려 했는가
- 목적 중심으로 간단 명확하게 작성

---

## 3.3 Progress

실제로 수행한 결과
반드시 Percent(%)가 포함되어야 함.

- Plan과 Progress의 차이를 통해 달성률과 병목을 분석 가능

---

## 3.4 Next

다음 주 혹은 다음 단계에서 이어서 수행해야 할 작업

---

## 3.5 Risk

현재 작업 진행에서 발생한 리스크 또는 잠재적 문제 요소

예:

- 저장 플로우에서 race condition 재발 가능성
- 특정 브라우저 레이아웃 깨짐 위험
- API 응답 지연 가능성

없으면 생략 가능

---

## 3.6 RiskLevel

리스크의 강도/중요도를 0–3 사이 값으로 표시한다.

| Level | 의미                            |
| ----- | ------------------------------- |
| 0     | 없음 / 영향 낮음                |
| 1     | 경미, 인지 필요                 |
| 2     | 주의 필요, 후속조치 요구        |
| 3     | 심각, 리소스/우선순위 조정 필요 |

---

# 4. Collaborators 필드

```
- Collaborators: 김정빈(pair), 조해용(waiting-on)
```

## 4.1 기본 형식

```
이름(relation)
```

여러 명은 쉼표로 구분한다.

---

## 4.2 relation 타입 (v1에서 2종)

| relation   | 의미                                          |
| ---------- | --------------------------------------------- |
| pair       | 함께 붙어서 공동 작업한 상태                  |
| waiting-on | 내 작업이 이 사람의 작업·확인을 기다리는 상태 |

### pair 상세

- FE–BE가 동시에 붙어서 스펙/작업을 맞춘 경우
- 기획–디자인이 실시간으로 플로우를 조정하는 경우
  > “이 작업은 혼자 한 것이 아니라 이 사람과 공동 기여한 작업”이라는 의미

### waiting-on 상세

- 스펙 확정 대기
- API 응답 대기
- 기획/디자인 등 다른 역할의 선행 작업 대기
  > “내 일을 막고 있는 원인(병목)을 명확히 드러내는 데이터”

---

# 5. 스냅샷 구조와 대시보드 활용

| 요소          | 활용 예                           |
| ------------- | --------------------------------- |
| domain        | 리소스/역량 분포 분석             |
| project       | 프로젝트별 업무 흐름 파악         |
| module        | 특정 기능 영역의 집중도·병목 파악 |
| topic         | 구체적인 업무 추적                |
| collaborators | 협업 패턴, 블로킹 관계, 병목 파악 |

해당 구조는 이후 AI 요약, 협업 패턴 분석, 임팩트 분석 등의 기반 데이터로 활용된다.

---

# 6. 요약

스냅샷 v1 포맷은 다음 네 가지 흐름을 기반으로 한다:

1. **domain → project → module → topic**  
   — 일의 위치를 계층적으로 구조화

2. **Plan → Progress → Next → Risk → RiskLevel**  
   — 실행 상황을 한눈에 파악하기 위한 핵심 필드

3. **Collaborators(name(relation))**  
   — 협업 및 의존 관계를 최소 입력으로 명확하게 표현

4. **확장성과 일관성을 모두 고려한 구조**  
   — 프로젝트, 모듈, relation 타입이 계속 추가될 수 있는 형태

---

위클리 스크럼 사이트의 파서/대시보드/AI 분석 기준에도 동일하게 적용된다.
